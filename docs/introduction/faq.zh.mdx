# 常见问题

## 入门问题

### 如何安装 Syl 编译器？

克隆项目仓库，在根目录运行 `cargo build --release`。编译完成后，可执行文件在 `target/release/sylc`。系统需要安装 Rust 工具链（rustc、cargo）。

### Syl 代码的文件扩展名是什么？

`.syl`。

### Syl 需要额外的运行环境吗？

不需要。编译器是静态链接的可执行文件。生成的 SystemVerilog 代码可以使用任何标准仿真工具处理。

### 如何编译一个 Syl 文件？

在终端运行 `sylc input.syl`。编译器输出 SystemVerilog 代码到标准输出。使用 `sylc input.syl -o output.sv` 输出到文件。

### 可以同时编译多个源文件吗？

可以。`sylc file1.syl file2.syl`。多个源文件属于同一个编译单元，可以互相引用。

## 语言问题

### `:=` 和 `next` 有什么区别？

`:=` 是连续赋值。赋值语句持续生效，右侧值变化时左侧立即更新。对应 Verilog 的 `assign`。

`next` 是寄存器下一状态赋值。值在时钟边沿更新到寄存器。对应 Verilog 的 `always @(posedge clk)` 中的非阻塞赋值。

### `signal` 和 `reg` 有什么区别？

`signal` 声明一个组合逻辑信号。它不存储状态，值由赋值表达式持续驱动。

`reg` 声明一个寄存器。它在时钟边沿采样输入值，保持到下一个时钟边沿。`reg` 需要配合 `reset` 指定复位值和复位信号。

### `let` 和 `signal` 有什么区别？

`let` 绑定一个名字，不创建硬件对象。`let x = expr` 只是给表达式 `expr` 起了一个别名，不产生任何硬件开销。

`signal` 创建一个实际的硬件信号。`signal x: Bit := expr` 在综合结果中对应一条物理连线。

### `map` 和 `fn` 有什么区别？

`map` 定义纯组合逻辑映射。`map` 内部不能使用寄存器或状态。`map` 在硬件中被展开为组合逻辑电路，没有面积开销。

`fn` 定义编译期执行的函数。`fn` 在编译期运行，不生成硬件。`fn` 的结果用于计算常量值。

### `cell` 和 `module` 有什么区别？

`cell` 是内联展开的硬件单元。`cell` 被实例化时，它的内部逻辑被展开到父模块中，不产生层次边界。

`module` 是独立的硬件模块。`module` 被实例化时，生成的 SV 代码中保持独立的模块层次。

### 枚举的变体如何引用？

使用 `EnumName.VARIANT_NAME` 语法。例如 `match color { Color.Red => ... }`。变体名必须使用枚举类型限定，编译器从表达式结构确定变体所属的枚举类型。

不支持和 Verilog 类似的裸变体名引用（如 `Red`）。

### 枚举可以指定编码值吗？

可以。枚举变体可以携带显式判别值：

```syl
enum State {
    Idle = 1,
    Busy = 3,
    Done = 7,
}
```

未赋值的变体沿用前一个值加 1。

### 枚举可以指定硬件宽度吗？

可以。在枚举名后加 `: UInt<N>` 指定表示宽度：

```syl
enum State: UInt<3> {
    Idle = 1,
    Busy = 3,
    Done = 7,
}
```

未指定宽度时，编译器按最大判别值推导最小必要 bit 宽度。

### 枚举支持 one-hot 或 flags 编码吗？

支持。通过 `@layout` 属性指定编码布局：

```syl
@layout(flags)
enum Access: Bits<4> {
    Read,
    Write,
    Exec,
    Admin,
}
```

`@layout(flags)` 自动分配 one-hot 值，支持 `|`、`&`、`^` 位运算组合。
`@layout(onehot)` 编码值分配相同，但语义上各状态互斥，不允许组合。

## 综合问题

### Syl 可以综合吗？

Syl 编译器输出 SystemVerilog 代码。生成的代码经过 Verilator 语法检查。你可以使用标准的综合工具（如 Yosys、Design Compiler、Vivado）对输出的 SV 代码进行综合。

### 生成的 SV 代码质量如何？

目前 Syl 生成的 SystemVerilog 代码风格比较直接：每个 Syl 模块对应一个 SV 模块，`:=` 赋值对应 `assign` 连续赋值，`next` 赋值对应时序逻辑。综合工具可以正常处理这种风格的代码。Syl 不做面积或时序优化。

### 生成的 SV 代码包含注释吗？

生成的 SV 文件顶部包含一行注释标识来源。模块内的信号和赋值不包含额外的注释。

### Syl 的编译速度如何？

对于小型设计（几个模块，几百行代码），编译时间在秒级。编译器目前没有针对大型设计做优化。

## 语言对比

### Syl 和 Chisel 比怎么样？

**架构差异。** Chisel 是内嵌在 Scala 中的领域专用语言，Syl 是一门独立的硬件描述语言。Chisel 的硬件生成在 Scala 运行时执行，Syl 的展开在编译器内部执行。

**实际影响。**
- **工具链。** Chisel 需要安装 sbt、JVM 和匹配的 Scala 版本，搭建项目需要数十分钟。Syl 编译器是单个静态链接的二进制文件，下载即可运行。
- **报错信息。** Chisel 的硬件错误经过 Scala 编译器 → FIRRTL 编译器两层翻译后才呈现给用户，错误信息混有 Scala 类型系统术语。Syl 的错误信息直接使用硬件术语，从第一个字符就是设计者熟悉的语言。
- **展开模型。** Chisel 的展开阶段（Scala 运行时）可以执行任意 Scala 代码，包括带副作用的操作。Syl 的编译期计算（`fn`、`const`）是纯函数式的，同一段代码在任何上下文产生相同结果。
- **学习曲线。** Chisel 要求先理解 Scala 的类型系统和隐式转换，然后才能上手硬件设计。Syl 只需要学一门专为硬件设计的语言。

**什么时候选 Chisel？** 项目规模大、团队已有 Scala 经验、需要利用成熟的 Scala 生态工具。**什么时候选 Syl？** 希望最小化工具链开销、希望编译错误直接说硬件术语、偏好无副作用的编译期展开。

### Syl 和 SpinalHDL 比怎么样？

**架构差异。** SpinalHDL 和 Chisel 类似，内嵌在 Scala 中。SpinalHDL 的硬件展开在 Scala 运行时执行，Syl 的展开在编译器内部执行。SpinalHDL 的 `Component` 对应 Syl 的 `cell`，`Area` 对应 `cell`（用 `inplace` 标记内联），`IMasterSlave` 对应 `view`。

**实际影响。**
- **类型系统。** SpinalHDL 的 `Bool`、`Bits`、`UInt`、`SInt` 是 Scala 库中的类型，错误信息经过 Scala 类型推断器的包装。Syl 的 `Bit`、`UInt<W>` 是语言原生类型，编译器的类型检查直接操作硬件语义。
- **接口方向。** SpinalHDL 的 `IMasterSlave` 方向在 Scala 运行时通过 `asMaster()` 方法调用设置。Syl 的 `view source/sink` 在语言层面声明，方向信息在编译期固定，不需要运行时代码。
- **层级控制。** SpinalHDL 的 `Component` 和 `Area` 运行时行为不同。Syl 的 `cell` 默认独立层次，`inplace` 标记内联展开，在语法层面区分。

**什么时候选 SpinalHDL？** 项目需要成熟的接口库（如 AXI、TileLink），需要丰富的外设模板。**什么时候选 Syl？** 希望接口方向在语言层面约束而非运行时配置，期望编译器直接从语法结构推导层次关系。

### Syl 和 Verilog 比怎么样？

Verilog 是业界标准，Syl 的一个输出目标是 Verilog 生态系统。Syl 在编译期做更多检查，减少仿真阶段的调试时间。

### Syl 和 SystemVerilog 比怎么样？

Syl 编译器的输出目标就是 SystemVerilog。Syl 并不是 SystemVerilog 的替代品，而是提供了一种更高抽象级别、更强静态检查的硬件描述方式，最终仍然使用 SystemVerilog 作为交付格式。

## 使用问题

### 需要学习 Rust 才能用 Syl 吗？

不需要。Syl 是一门独立的语言，和 Rust 没有关系。编译器的实现使用 Rust，但使用者不需要了解 Rust。

### 可以在 Windows 上使用 Syl 吗？

可以。Syl 编译器使用 Rust 编写，支持 Windows、macOS 和 Linux。

### Syl 支持模块级验证吗？

Syl 本身不提供仿真或验证功能。编译器生成 SystemVerilog 代码后，你可以使用现有的 UVM 验证环境或 SV 仿真工具对生成的代码进行验证。

### Syl 支持 UVM 吗？

不支持。编译器直接生成 RTL 级别的 SystemVerilog。UVM 验证环境需要手动编写。

### 可以在 Syl 中使用现有的 Verilog IP 吗？

可以通过 `extern cell` 声明外部 IP 的端口签名。编译器根据签名生成正确的实例化和连接。签名的准确性由使用者保证。

## 错误处理

### 编译器报告"类型不匹配"怎么办？

检查赋值两侧的位宽和类型是否一致。`Bit` 和 `UInt<1>` 是不同的类型。如果使用泛型参数，检查参数值是否在预期范围内。

### 编译器报告"多驱动冲突"怎么办？

检查同一个信号是否被多次赋值。每个 `signal` 或 `out` 端口在整个模块中只能被赋值一次。`select` 或 `match` 表达式中的多个分支赋值同一个目标是允许的。

### 编译器报告"未知名字"怎么办？

检查名字是否拼写正确，是否被正确导入。跨文件引用需要使用 `use` 导入。模块内部的信号名字不能和端口名字重复。

## 贡献

### 如何为 Syl 做贡献？

查阅项目仓库的贡献指南。目前特别需要以下方面的帮助：使用 Syl 实际设计硬件并反馈问题，编写标准库组件，改进诊断信息。

### 如何报告 bug？

在项目仓库提交 GitHub issue。请包含出错的 Syl 代码、预期行为、实际行为、编译器版本和环境信息。

### 如何请求新功能？

在项目仓库提交 GitHub issue。说明使用场景和需求，帮助理解为什么这个功能对你有用。
