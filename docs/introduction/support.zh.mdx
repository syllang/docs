# 支持

本文列出在遇到问题时可以通过哪些渠道获取帮助。

---

## 自行排查

在寻求帮助前，可以先尝试以下步骤：

1. 检查代码中是否有未定义的符号。`use` 导入的路径是否正确？模块名字是否拼写正确？
2. 检查端口方向是否正确。`in` 端口不能在模块内部赋值，`out` 端口不能在模块内部读取。
3. 检查类型是否匹配。赋值两侧的位宽是否一致？如果使用泛型参数，参数值是否在有效范围内？
4. 检查 `select` 和 `match` 表达式是否覆盖了所有分支。
5. 检查 `reg` 声明是否有 `reset`。寄存器如果没有复位信号，编译器可能会报错。
6. 检查 `let` 和 `signal` 的使用是否正确。`let` 只是别名，不创建硬件对象。`signal` 创建硬件信号。

编译器的错误信息会包含出错的源码位置和错误类型。将错误信息和相关代码一起搜索，通常能找到相似的案例。

## 理解编译器输出

运行 `sylc` 时，输出包含两部分：

**诊断信息。** 错误和警告会输出到标准错误（stderr）。每条诊断包含严重级别（error 或 warning）、错误代码、描述信息和源码位置。

```
error[E_MIDDLE_TYPE_MISMATCH]: expected `UInt<8>`, found `Bit`
  ┌─ input.syl:2:9
  │
2 │     y := x
  │         ^ expected `UInt<8>`, found `Bit`
```

**SystemVerilog 代码。** 编译成功后，SV 代码输出到标准输出（stdout）。使用 `-o` 参数输出到文件。

## 文档

首先查阅项目文档。`docs/` 目录包含以下内容：

- **语言基础**：类型系统、语法、接口、视图
- **编译流水线说明**：了解编译器如何处理代码
- **静态检查规则**：了解编译器检查哪些规则
- **迁移指南**：从 Verilog、Chisel、SpinalHDL 迁移
- **参考**：CLI 使用、语法定义、诊断代码列表

文档目录按功能组织：

| 章节 | 内容 |
|------|------|
| introduction | 语言概览、FAQ、入门 |
| language | 类型系统、语法、语义 |
| patterns | 常见设计模式 |
| reference | CLI、语法、诊断码 |
| tooling | 编译器、LSP、SV 输出 |

## GitHub Issues

在项目仓库提交 issue：

- **Bug 报告。** 编译器崩溃、错误诊断不正确、生成的 SV 代码无法综合。请附上最小可复现示例。
- **功能请求。** 语言特性、标准库组件、工具链改进。请说明使用场景。
- **设计讨论。** 语言设计决策的讨论。请在 issue 标题前加 `[design]` 标记。

提交 issue 前，先搜索是否存在已有 issue。搜索时使用相关的错误代码（如 `E_MIDDLE_TYPE_MISMATCH`）或功能名称（如 `Stream`、`Stage`）。

## 贡献代码

查看 CONTRIBUTING.md 了解贡献流程。目前特别期待以下方向的贡献：

- **使用 Syl 设计硬件。** 在实践中发现编译器的不足和 bug。
- **编写标准库组件。** 扩展标准库的总线接口和常用组件。
- **改善诊断信息。** 让编译器的错误信息更易于理解。
- **示例和教程。** 帮助新手更快上手。

## 项目状态跟踪

关注以下内容了解最新进展：

- CHANGELOG.md：每个版本的变更记录，包含新功能、bug 修复和破坏性变更
- ROADMAP.md：中长期的规划和优先级

## 反馈语言设计

Syl 处于早期阶段，语言设计决策欢迎反馈。如果你发现：

- 某个语言特性难以使用
- 某个编译器检查过于严格或不够严格
- 某个错误信息难以理解
- 缺少某个常用硬件设计模式的支持

请在 issue 中描述你的使用场景和遇到的问题。## 社区和讨论

Syl 目前还没有正式的社区论坛。以下渠道可以交流和讨论：

- **GitHub Issues**：技术问题、bug 报告、功能请求
- **GitHub Discussions**（如果已启用）：开放讨论和问答

## 常见问题排查顺序

如果你的代码编译失败，按以下顺序排查效率最高：

1. 查看第一个错误。编译器可能报告多个错误，但后续错误经常是第一个错误导致的连锁反应。修复第一个错误后重新编译。

2. 关注错误代码。错误信息中包含形如 `E_MIDDLE_TYPE_MISMATCH` 的错误代码。在文档的诊断索引中搜索这个代码，可以找到详细的解释和示例。

3. 缩小问题范围。创建一个最小可复现示例：去掉不相关的模块和信号，只保留触发错误的最少代码。

4. 检查修改历史。如果代码之前可以编译，最近修改了哪些部分？回退修改，确认问题是否由最新修改引入。

具体的使用场景描述比抽象的建议更有帮助。
