# 快速开始

本文带你在 5 分钟内从零走到第一个编译通过的 Syl 设计。先看一段 Syl 代码长什么样，然后安装编译器、编译一个计数器例子、看懂输出和错误信息。最后给你几条学习路径。

---

## Syl 代码长什么样？

下面是一个完整的 Syl 模块。你可以先看一眼，有个印象：

```syl
cell Counter<W: Nat = 16, D: Domain>(
    clk: in Clock<D>,
    rst: in Reset<D>,
    enable: in Bit,
    clear: in Bit,
) -> value: UInt<W> {
    reg count: UInt<W> reset(rst, 0)

    signal count_next: UInt<W> :=
        select priority {
            clear => 0,
            enable => count + 1,
            default => count,
        }

    next count := count_next
    value := count
}
```

这段代码定义了一个带参数的计数器模块。`<W: Nat = 16>` 表示位宽可配置，默认 16 位。`cell` 是 Syl 中定义硬件模块的关键字。`reg` 声明寄存器，`signal` 声明组合逻辑信号，`next` 指定寄存器的下一状态。

如果你暂时看不懂细节——没关系。安装完编译器后，我们逐行拆解它。

---

## 安装 Syl 编译器

Syl 编译器发布在 crates.io 上，通过 cargo 直接安装：

```bash
cargo install sylc
```

安装完成后，sylc 已加入 PATH，在终端直接使用：

```bash
sylc --help
```

你应该看到 sylc 的使用帮助，包括输入文件路径、输出选项等参数。

如果你还没有 Rust 工具链，先安装 Rust：

```bash
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
source "$HOME/.cargo/env"
rustc --version
```

### 从源码编译

如果你需要自己编译（比如调试编译器的行为），克隆仓库后运行：

```bash
git clone https://github.com/syllang/syl
cd syl
cargo build --release
```

首次编译需要下载依赖，大约 3 ~ 5 分钟。完成后编译器在 `target/release/sylc`。

## 编译第一个 Syl 文件

Syl 仓库自带例子。运行计数器例子：

```bash
sylc examples/mvp/counter.syl
```

如果编译成功，终端输出 SystemVerilog 代码。这是 Syl 编译器的产出——Syl 不直接生成网表，而是生成 Verilog 行业标准格式，供下游综合工具使用。你可以在终端中看到类似这样的输出：

```verilog
// Generated by sylc.

module Counter(
    input  wire         clk,
    input  wire         rst,
    input  wire         enable,
    input  wire         clear,
    output wire [15:0]  value
);
    reg [15:0] count;

    always @(posedge clk or posedge rst) begin
        if (rst)
            count <= 16'd0;
        else begin
            if (clear)
                count <= 16'd0;
            else if (enable)
                count <= count + 16'd1;
            else
                count <= count;
        end
    end

    assign value = count;
endmodule
```

现在回头看那一段 Syl 源码，逐行解释：

**模块声明部分：**

- `cell Counter<W: Nat = 16, D: Domain>`：`cell` 声明一个硬件模块。`<W: Nat = 16>` 声明一个类型参数 W，它必须是自然数，默认值 16。`<D: Domain>` 声明一个时钟域参数，用来连接时钟和复位信号。如果你要在同一芯片上使用两个不同时钟域，域参数让编译器检查信号不会跨域错误连接。
- `clk: in Clock<D>`：时钟输入端口。`Clock<D>` 的 D 表示这个时钟所属的域。如果两个模块使用不同的 D 值，它们的时钟信号不能混连。
- `rst: in Reset<D>`：复位输入端口。`Reset<D>` 的 D 必须和 clk 的 D 一致，否则编译器报错。这个检查避免了"把 A 时钟域的复位信号接到 B 时钟域的模块"这类布线错误。
- `enable: in Bit`、`clear: in Bit`：一位宽的输入端口，值只能是 0 或 1。
- `value: out UInt<W>`：W 位宽的无符号整数输出端口。W 默认是 16，所以默认是 16 位。

**模块体部分：**

- `reg count: UInt<W> reset(rst, 0)`：声明一个 W 位宽的寄存器。`reg` 表示这是一个时序元件。`reset(rst, 0)` 表示同步复位：当 rst 信号拉高时，count 复位到 0。
- `signal count_next: UInt<W> := select priority { ... }`：声明一个组合逻辑信号。`signal` 没有状态保持能力，它的值由右侧表达式即时计算。`:=` 是赋值运算符。`select priority` 按优先级从上到下检查条件，匹配第一个为真的分支。如果 `clear` 和 `enable` 同时为 1，因为 `clear` 写在前面，它胜出。
- `next count := count_next`：将寄存器的下一状态连接到 `count_next` 这个信号。每个时钟上升沿，count 的值更新为 count_next 的当前值。
- `value := count`：将寄存器的当前值连接到输出端口。

**为什么 Syl 要把寄存器的行为拆成 `signal` 和 `next` 两句？** `signal count_next` 定义下一状态的组合逻辑——它只是一个加法和一个条件选择。`next count := count_next` 定义寄存器的更新规则——每个时钟沿将组合逻辑的结果锁存到寄存器中。这种分离让编译器可以独立检查：组合逻辑部分有没有冲突（比如两个不同的值同时驱动 count_next）？时序逻辑的驱动是否完整（count 有没有被遗漏的赋值）？

## 一个更复杂的例子

`examples/mvp/comb_alu.syl` 展示了枚举类型、信号组合和纯组合函数：

```syl
enum AluOp {
    Add,
    Xor,
    PassA,
}

bundle AluReq<W: Nat> {
    op: AluOp,
    a: UInt<W>,
    b: UInt<W>,
}

bundle AluRsp<W: Nat> {
    data: UInt<W>,
    zero: Bit,
}

map alu_data<W: Nat>(req: AluReq<W>) -> UInt<W> =
    match req.op {
        .Add => req.a + req.b,
        .Xor => req.a xor req.b,
        .PassA => req.a,
    }

cell CombAlu<W: Nat>(req: in AluReq<W>) -> rsp: AluRsp<W> {
    signal data: UInt<W> := alu_data<W>(req)

    rsp := AluRsp<W> {
        data: data,
        zero: data eq 0,
    }
}

cell CombAlu32(req: in AluReq<32>) -> rsp: AluRsp<32> {
    signal inner: AluRsp<32> :=
        place CombAlu<32>(req: req)

    rsp := inner
}
```

逐层拆解：

**`enum AluOp`** 定义一个枚举类型，有三个变体：`Add`、`Xor`、`PassA`。枚举类型让操作码在类型层面是安全的。如果代码里把 `Add` 赋值给一个不是 `AluOp` 类型的信号，编译器拒绝。

**`bundle AluReq<W: Nat>` 和 `bundle AluRsp<W: Nat>`** 将多个信号组合为一个类型。`AluReq` 包含操作码 `op` 和两个操作数 `a`、`b`。`AluRsp` 包含计算结果 `data` 和零判断标志 `zero`。`<W: Nat>` 让这两个 bundle 的位宽可配置。和 Verilog 的结构体相比，Syl 在编译期检查 bundle 的每个字段类型是否匹配——传错字段类型或位宽，编译器直接报错。

**`map alu_data`** 定义纯组合逻辑函数。`map` 没有寄存器、没有内部信号，只有输入到输出的组合逻辑映射。`match req.op { .Add => ... }` 根据操作码选择计算方式。`=>` 的意思是"匹配到这个模式时，执行右侧的表达式"。因为 `map` 是纯组合逻辑，编译器可以充分检查它的类型安全——每个分支的返回值类型必须一致。

**`cell CombAlu`** 将 map 包装为硬件模块。`signal data := alu_data<W>(req)` 实例化 map，将输出赋给内部信号。`rsp := AluRsp<W> { data: data, zero: data eq 0 }` 通过字段名构造一个 bundle，赋值给输出端口。`data eq 0` 是 Syl 的相等比较运算，结果为 Bit 类型（0 或 1）。

**`cell CombAlu32`** 展示参数化模块的实例化。`place CombAlu<32>(req: req)` 将 W 固定为 32，生成一个 32 位宽的 ALU 实例。`place` 是 Syl 的实例化关键字，类似 Verilog 中在模块内部实例化另一个模块。

编译这个文件：

```bash
sylc examples/mvp/comb_alu.syl
```

输出的 SystemVerilog 会包含两个顶层模块：`CombAlu`（参数化版本）和 `CombAlu32`（参数固定的版本）。每个模块的端口声明、数据类型和内部连接都由编译器从 Syl 源码推导生成。

注意 `CombAlu32` 中 `place` 关键字的使用。`place` 是实例化关键字，在模块内部放入另一个模块的实例。它和 `signal` 不同：`signal` 声明一个组合逻辑信号，值由右侧表达式决定；`place` 放入一个完整的子模块实例，有自己的端口和内部结构，编译器最终在输出中生成对应的子模块声明。

## 保存输出到文件

用 `-o` 参数把编译结果保存到文件：

```bash
sylc examples/mvp/counter.syl -o counter.sv
```

这条命令把 SystemVerilog 代码写入 `counter.sv` 文件。之后你可以把这个文件交给下游工具——比如用 Verilator 做仿真，或者用 Yosys 做综合。

## 读懂编译错误

Syl 的错误直接使用硬件术语，不经过宿主语言翻译。下面是三种最常见的情况。

### 多驱动冲突

```syl
cell DoubleDrive() -> y: Bit {
    y := 0
    y := 1
}
```

编译输出：

```
error[E_DRIVER_OVERLAP]: multiple drivers for signal 'y'
  ┌─ input.syl:4:5
  │
2 │   y := 0
  │   ----- first driver here
3 │   y := 1
  │   ----- second driver here
```

错误信息告诉你：信号 `y` 被两个来源驱动，分别是第 2 行的 `y := 0` 和第 3 行的 `y := 1`。编译器知道一个信号只能有一个驱动源。多个条件赋值请用 `select` 语句。

### 类型不匹配

```syl
cell Bad(x: in Bit, y: out UInt<8>) {
    y := x
}
```

编译输出：

```
error[E_TYPE_MISMATCH]: expected 'UInt<8>', found 'Bit'
  ┌─ input.syl:3:10
  │
2 │   y: out UInt<8>
3 │   y := x
  │   ----- expected UInt<8>
```

端口 `y` 是 8 位宽，`x` 是 1 位宽。编译器拒绝这种不匹配的赋值。修复方式是确保两侧位宽一致，或者显式做位宽扩展。

### 方向违规

```syl
cell Bad(x: in Bit) -> z: Bit {
    z := not x
}
```

编译输出：

```
error[E_CAPABILITY]: cannot read signal 'x' of direction 'in'
  ┌─ input.syl:3:12
  │
2 │   x: in Bit
3 │   z := not x
  │         ^ cannot read 'in' port
```

`in` 方向端口的值只能由外部提供，模块内部不能读取它。如果你需要同时对外部读取和驱动一个端口，使用 `inout` 方向。

所有诊断码的完整解释在[诊断代码参考](/zh/docs/reference/diagnostics)中。每个诊断码都是一个链接，点进去可以看到更详细的说明和修复建议。

## 注意事项

Syl 处于早期开发阶段。标准库还不完整，一些高级功能还在实现中。但这不影响你用现在的版本评估设计理念和构建中小型原型。

如果在使用中遇到问题，在 Syl 仓库的 Issues 页提交。提交时请附上出错的 Syl 代码和编译器的完整输出。
