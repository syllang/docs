# 版本与兼容性

本文说明 Syl 的版本号规则和兼容性保证。了解这些规则有助于判断升级编译器的风险。

---

## 版本号规则

Syl 使用语义化版本号：`MAJOR.MINOR.PATCH`。

**MAJOR（主版本号）。** 不兼容的 API 或语言变更。升级主版本号可能需要修改 Syl 代码。

**MINOR（次版本号）。** 向下兼容的功能新增。升级次版本号可以保留现有功能。

**PATCH（修订号）。** 向下兼容的问题修复。升级修订号不会引起代码变更。

当前 Syl 处于 `0.x.x` 阶段。在 `0.x` 阶段，次版本号变更可能引入不兼容变化。

## 语言兼容性

### 编译器输出

Syl 编译器生成 SystemVerilog 代码。生成的 SV 代码的行为应该在不同版本之间保持一致。如果存在行为不一致，视为编译器 bug。

### 语言语法

在 `0.x` 阶段，语法可能变化。变化包括：

- 新增语法特性（可能影响关键字的使用）
- 修改错误检查规则（现有代码可能触发新的错误）
- 废弃某些语法形式（编译器给出弃用警告）

重大的语法变化会在 Changelog 中说明。

### 标准库

标准库的接口在 `0.x` 阶段可能变化。变化包括：

- 新增组件
- 修改现有组件的接口参数
- 移除废弃的组件

## 升级策略

### 小版本升级

阅读 Changelog，确认变更内容。运行测试确认功能正常。

### 大版本升级

1. 阅读 Changelog 了解破坏性变更
2. 运行编译器检查代码中的弃用警告
3. 根据更新说明修改代码
4. 运行测试确认行为一致

### 预览版本

预览版本（alpha、beta、rc）用于测试。预览版本之间可能有破坏性变更，不建议用于生产项目。

## 兼容性保证

### 编译期保证

同一版本的编译器对相同代码应该产生相同行为。如果不同编译器调用产生不同结果，视为 bug。

### 输出保证

编译器生成的 SystemVerilog 代码应该在语义上等价于输入的 Syl 代码。如果输出代码的行为和输入描述不一致，视为 bug。

### 诊断保证

对于错误的 Syl 代码，编译器应该报告诊断信息。诊断信息的具体文本可能在版本之间变化，但诊断的存在性应该保持一致。
