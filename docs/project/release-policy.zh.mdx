# 发布策略

本文说明 Syl 编译器的发布流程、节奏和渠道。

---

## 发布节奏

Syl 采用不定期发布策略。发布取决于功能完成度和质量门状态。

**主要发布。** 包含新功能或破坏性变更。发布前会经过完整的测试流程：架构合约测试、一致性测试、Verilator 烟雾测试。主要发布有 Changelog 记录变更内容。

**补丁发布。** 包含 bug 修复。补丁发布不引入新功能或破坏性变更。补丁发布可以更快地进行，不需要完整的功能测试。

## 发布流程

1. **功能冻结。** 确定发布范围，停止新功能开发。
2. **测试阶段。** 运行完整的测试套件，包括架构合约和一致性测试。
3. **修复阶段。** 修复测试中发现的问题。
4. **发布候选。** 创建发布候选版本，进行集成测试。
5. **发布。** 标记版本、更新 Changelog、发布 GitHub Release。

## 发布渠道

**GitHub Releases。** 主要的发布渠道。每个版本包含预编译的二进制文件（如果需要）和源码归档。

**crates.io。** 基础 crate（如 `syl_span`）可能会发布到 crates.io。编译器的核心 crate（`syl_syntax`、`syl_sema` 等）在稳定之前可能不发布。

## 质量门

每次发布前必须通过以下检查：

1. 所有测试通过
2. 架构合约测试通过
3. 文档中的 mermaid 图语法正确
4. 文档阅读时间检查通过（6-20 分钟）
5. Verilator 烟雾测试通过（如果 Verilator 可用）
6. Clippy lint 检查通过

## 版本历史

完整的版本历史在 Changelog 中。每个版本记录：

- 新增功能
- 破坏性变更
- Bug 修复
- 文档变更
- 已知问题
