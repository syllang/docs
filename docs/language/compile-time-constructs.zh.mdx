# 编译期构造

Syl 在编译期执行部分代码。这些代码不生成硬件，它们的结果用于确定硬件参数。编译期构造包括 `const` 常量、`fn` 函数、`if` 条件分支和 `for` 循环。

---

## const

`const` 声明一个编译期常量。它的值在编译期确定，不能修改。

### 基本用法

```syl
const WIDTH: Nat = 8
const DEPTH: Nat = 16
const DEBUG: Bool = true
```

常量可以参与编译期计算：

```syl
const WORDS: Nat = 4
const TOTAL_BITS: Nat = WORDS * WIDTH
```

### 常量作为类型参数

常量的主要用途是作为类型参数的值：

```syl
const BUF_SIZE: Nat = 64

cell Buffer(
    data_in: in UInt<BUF_SIZE>,
    data_out: out UInt<BUF_SIZE>,
)
```

### 常量表达式

常量的初始值可以是表达式，只要表达式在编译期可求值：

```syl
const ADDR_BITS: Nat = clog2(16)    // 调用编译期函数
const MASK: Nat = (1 << 8) - 1       // 算术运算
const IS_DEBUG: Bool = (WIDTH > 32)  // 比较运算
```

## fn

`fn` 定义编译期执行的函数。它支持参数、条件和循环。

### 基本定义

```syl
fn clog2(x: Nat) -> Nat {
    var n: Nat = 0
    var p: Nat = 1

    while p < x {
        p = p << 1
        n = n + 1
    }

    return n
}
```

### 调用

```syl
const W: Nat = clog2(64)     // W = 6
const ADDR_W: Nat = clog2(16) // ADDR_W = 4
```

### fn 中的变量

`fn` 内部使用 `var` 声明可变变量。变量只在 `fn` 的执行过程中存在，不生成硬件。

```syl
fn sum_to(n: Nat) -> Nat {
    var result: Nat = 0
    var i: Nat = 1
    while i <= n {
        result = result + i
        i = i + 1
    }
    return result
}
```

### fn 的限制

`fn` 只能操作编译期类型（`Nat`、`Bool`）。不能直接操作硬件信号（`Bit`、`UInt<W>`）。如果需要在编译期分析硬件参数，`fn` 是合适的工具。如果需要处理硬件信号，使用 `map`。

## 编译期 if

编译期 `if` 的条件必须是 `Bool`（编译期布尔值）。条件为 `true` 时保留 then 分支，为 `false` 时保留 else 分支。

```syl
const HAS_TIMEOUT: Bool = true

cell Controller(
    data_in: in UInt<8>,
    data_out: out UInt<8>,
) {
    if HAS_TIMEOUT {
        signal timeout: Bit
        // 带超时的逻辑
    } else {
        // 不带超时的逻辑
    }
}
```

如果 `HAS_TIMEOUT` 是 `false`，编译器完全不生成 `timeout` 相关的硬件。这比在运行时检查条件效率更高。

### if 用于参数验证

```syl
cell Buffer<W: Nat>(data: in UInt<W>) {
    if W == 0 {
        compile_error("Buffer width must be greater than 0")
    }
    if W > 64 {
        compile_error("Buffer width must not exceed 64")
    }
}
```

参数不合法时，编译器在编译期报错。

## 编译期 for

编译期 `for` 循环在编译期展开。循环次数必须是编译期常量。

```syl
signal values: [4] UInt<8> := for i in 0..4 {
    inputs[i] + 1
}
```

展开后等价于：

```syl
signal values_0: UInt<8> := inputs[0] + 1
signal values_1: UInt<8> := inputs[1] + 1
signal values_2: UInt<8> := inputs[2] + 1
signal values_3: UInt<8> := inputs[3] + 1
```

### 在模块体中使用 for

```syl
cell ArrayAdd<N: Nat, W: Nat>(
    inputs: in [N] UInt<W>,
    outputs: out [N] UInt<W>,
) {
    for i in 0..N {
        outputs[i] := inputs[i] + 1
    }
}
```

`N` 是泛型参数。实例化 `ArrayAdd<4, 8>` 时生成 4 个加法器。

### 嵌套 for

```syl
for i in 0..4 {
    for j in 0..4 {
        matrix[i][j] := i * 4 + j
    }
}
```

嵌套 `for` 在内层和外层分别展开。总展开次数是内外次数的乘积。

## compile_error

`compile_error(message)` 在编译期报错。它通常和编译期 `if` 配合使用，用于参数验证。

```syl
cell FIFO<W: Nat, D: Domain>(
    clk: in Clock<D>,
    rst: in Reset<D>,
    data_in: in UInt<W>,
    data_out: out UInt<W>,
) {
    if W == 0 {
        compile_error("FIFO width must be greater than 0")
    }
    if W > 1024 {
        compile_error("FIFO width 1024 exceeds implementation limit")
    }
}
```

## 设计和测试中的使用

编译期构造在设计验证中也有用。下面是一个常见的模式：用 `fn` 实现参数检查，避免非法参数导致综合错误。

```syl
fn valid_config(width: Nat, depth: Nat) -> Bool {
    if width == 0 { return false }
    if depth == 0 { return false }
    if width > 64 { return false }
    return true
}

const CFG_OK: Bool = valid_config(8, 16)

cell Buffer<W: Nat, D: Nat>(
    data_in: in UInt<W>,
    data_out: out UInt<W>,
) {
    if not CFG_OK {
        compile_error("invalid buffer configuration")
    }
}
```

参数检查在模块实例化时执行。如果参数不合法，编译立即失败。

## 性能考虑

编译期计算对编译速度有影响。复杂的 `fn` 函数（嵌套循环、深度递归）会增加编译时间。

不过这种影响通常是可接受的，因为编译期代码只执行一次。即使 `fn` 执行了数百万次操作，编译时间增加也只是毫秒级别。

另外，编译期 `for` 循环展开会增加生成的硬件面积。`for i in 0..N` 展开为 N 份独立的逻辑。如果 N 很大（比如 1024），生成的门数也会很大。

## 常见错误

### 编译期类型和硬件类型混用

```syl
# fn add(x: UInt<8>) -> UInt<8> { ... }  // 错误：fn 不能操作硬件类型
```

`fn` 只接受编译期类型。需要处理硬件信号的逻辑使用 `map`。

### for 范围不是常量

```syl
# for i in 0..variable { }   // 错误：variable 不是编译期常量
```

`for` 的循环次数必须在编译期确定。

### 常量值超出硬件范围

```syl
const W: Nat = 256
signal s: UInt<8> := W       // 错误：256 超出 UInt<8> 范围
```

常量值如果用于硬件信号，不能超出信号类型的表示范围。
